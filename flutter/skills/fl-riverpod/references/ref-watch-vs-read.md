## Riverpod: ref.watch in build, ref.read in callbacks

`ref.watch` registers a live subscription and rebuilds the widget on state change — it
belongs in `build`. `ref.read` snapshots the current value once with no subscription — it
belongs in callbacks and event handlers. Swapping them causes stray subscriptions
(watch-in-callback) or stale UI (read-in-build).

**Incorrect:**
```dart
// watch inside a callback — creates a dangling subscription
ElevatedButton(
  onPressed: () => ref.watch(alarmListProvider.notifier).acknowledgeAll(),
  child: const Text('Ack'),
)

// read inside build — returns stale state, no rebuild on change
Widget build(BuildContext context, WidgetRef ref) {
  final alarms = ref.read(alarmListProvider); // stale!
  return Text('${alarms.length} alarms');
}
```

**Correct** (`alarmListProvider` is generated by `@riverpod`):
```dart
Widget build(BuildContext context, WidgetRef ref) {
  // subscribe in build — widget rebuilds when list changes
  final alarms = ref.watch(alarmListProvider);
  return ElevatedButton(
    // snapshot in callback — no subscription, just triggers action
    onPressed: () => ref.read(alarmListProvider.notifier).acknowledgeAll(),
    child: Text('Ack ${alarms.length}'),
  );
}
```

Review heuristic: search for `ref.watch` outside a `build` method — every hit is a bug.

Ref: https://riverpod.dev/docs/concepts/reading  
Ref: https://riverpod.dev/docs/migration/from_riverpod_2_0_0_to_3_0_0

### ref.mounted after await (3.0 — first-class guard)

After every `await` in a notifier method, guard with `if (!ref.mounted) return;` before
touching `ref` or `state`. The provider may be disposed between suspension and resume.

```dart
// inside @riverpod class AlarmList extends _$AlarmList { ... }
Future<void> acknowledgeAll() async {
  state = const AsyncValue.loading();
  final result = await ref.read(alarmRepositoryProvider).acknowledgeAll();
  if (!ref.mounted) return; // ← required after every await in 3.0
  state = AsyncValue.data(result);
}
```

### ref.listen variants

- `ref.listen(provider, cb)` — keeps the provider alive while active.
- `ref.listen(provider, cb, weak: true)` — listener that does NOT keep the provider alive;
  useful for side-effects that shouldn't prevent disposal.
- `ref.listenSelf(...)` moved to `Notifier.listenSelf` in 3.0 — call it as `this.listenSelf(...)` (or just `listenSelf(...)`) inside a notifier class, not on `ref`.
