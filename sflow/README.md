# sflow — unified spec-driven development workflow

`sflow` (formerly `specflow` — renamed so the bundle never collides with a project's own
`specflow/` folder or `/spec-*` command set) is a stack-neutral, spec-driven development workflow
that ensures a spec marked "Completed" provably means *the stated behavior is verified in an
independently testable unit*. It binds every acceptance criterion to a named, outcome-asserting
test **at authoring time** and gates the architecture **at phase exit**.

## The layers

```
project-hosted templates ── PHASE MACHINES: the specflow workflow YAMLs vendored in the TARGET
         │                    project — specflow/src/workflows/<variant>.yaml, overridable at
         │                    .specflow/workflows/<variant>.yaml. The bundle ships NO templates.
         │ executed via
sflow/commands/          ── PROCESS: 8 stack-neutral /sf-* commands (procedure guides;
         │                    no skill names, no driver knowledge — user-invocable one by one)
         │ generated by
skills/sf-react-workflow       ── WORKFLOW GENERATORS (repo root, self-contained real dirs):
skills/spec-react-workflow        translate a template into the spec's generated workflow.yaml —
         │                    commands + concrete OAC React skills + gates + exitWhen; React-only,
         │                    binding them directly to phases in references/phase-map.md
         │                    (each generator holds its own copy; zero symlinks between skills)
         │ driven by
agents/                  ── ORCHESTRATORS: two flow drivers — my-specflow-driver,
         │                    oac-specflow-driver; each embeds its Setup (run by its
         │                    initialPrompt), drives the phases, enforces gates, delegates.
         │                    Plus three BOUND WORKERS both drivers spawn — code-auditor-agent
         │                    (owns the atlas: build/query/extend), react-test-agent, react-impl-agent:
         │                    stack-neutral roles whose frontmatter preloads the profile's
         │                    skills (`skills:` — the language specifics) + audit-code-flows,
         │                    and pins model/effort/tools; bodies carry fences + return
         │                    contract only, never skill content
profiles/skills/         ── IMPLEMENTATION: self-contained problem-solvers with references/
profiles/rules/          ── ALWAYS-ON: short, path-gated topic files (no code listings)
```

**Domain skills and rules know nothing about sflow.** No domain skill or rule names a phase, an
`/sf-*` command, or the spec dir — each is a standalone problem-solver usable outside the
workflow; anything one needs (spec dir, artifact paths, output dir) is passed in by the caller.
The single deliberate exception is the **workflow generators** (`sf-react-workflow` /
`spec-react-workflow`) — they *are* the binding layer: each owns the template→workflow
translation for its flow, and the phase→skill binding table plus exitWhen lines
(`references/phase-map.md`) live under its own `references/` as **real files** (the two
generators hold duplicated copies — no symlinks between skill dirs). They are React-only,
binding the OAC React skills directly to phases; a Flutter/other stack would ship a parallel
`*-<tech>-workflow` generator. They write the generated `workflow.yaml` the driver orchestrates
from.

## Workflow templates — project-hosted

The bundle ships no workflow templates. `/sf-init` embeds the **feature** phase list
directly (feature is the only sflow variant for now — no template lookup). The generators
read the specflow templates from the **target project**, in this order:

1. `specflow/src/workflows/<variant>.yaml` — the specflow repo vendored in the project.
2. `.specflow/workflows/<variant>.yaml` — project override.

Neither found → STOP: the project must vendor specflow (or the user supplies the template path).

Variants and their phase ids (exact `.meta.yaml` `phase_status` keys):

| Variant | Phases |
|---|---|
| `feature` (recommended) | preflight → requirements → clarify → design → tasks → taskstoissues → implement → spec-qa |
| `brownfield` (deprecated upstream) | analysis → requirements → design → tasks → implement → spec-qa |
| `bugfix` (deprecated upstream) | analysis → tasks → implement → spec-qa |
| `quickfix` (deprecated upstream) | describe → implement → spec-qa |

Per phase a template carries `id`, `approval` (`human|auto|skip`), `required`, `inputs`,
`outputs` — plus `generator`/`executor` hints, `validators`, and `hooks`, which the generators
ignore. Phase ids are kept verbatim in the generated `workflow.yaml` (`spec-qa` stays
`spec-qa`) — `.meta.yaml` consistency rules over aesthetics.

## Starting a spec — init, then generate

Starting a spec is driver-led. The driver's embedded **Setup** section (run by its
`initialPrompt`):

1. **Worktree check** — write nothing until the repo is confirmed a worktree.
2. **`/sf-init`** — scaffolds `.specflow/specs/<name>/` + `.meta.yaml` ONLY (phase ids read
   from the variant's project-hosted template, all `pending`). No `workflow.yaml` yet.
3. **WAIT for the user's instructions**, then **`/sf-react-workflow <variant>`** — the sole
   writer of `workflow.yaml`: it translates the template (skills, prompts, gates, exitWhen),
   verifies `workflow:` + phase ids against `.meta.yaml`, and writes
   `.specflow/specs/<name>/workflow.yaml`. Then the Drive Loop begins.

Every later command and the driver read that generated file — never the template — so a spec is
self-contained and resumable.

### The generated workflow.yaml — schema

Both generators emit exactly this shape; the drivers hold no other process knowledge:

```yaml
workflow: feature | brownfield | bugfix | quickfix   # MUST match .meta.yaml `workflow:`
description: <workflow-description>
phases:
  - id: <phase-id>          # MUST match the .meta.yaml phase_status keys
    skills:
      - <skill-name>        # every skill the phase invokes; the /sf-* command is the first
    prompt: <text>          # self-contained instruction naming each skill above verbatim
    inputs:
      - <input-name>
    outputs:
      - <output-name>
    gate: human | auto
    exitWhen: <exit-condition>
```

There is **no** `command` field — a phase's `/sf-*` command is just the first entry in `skills`,
woven verbatim into `prompt`; the driver runs `prompt` directly and never dispatches a command.
Driver-led phases (`analysis`, `describe`) simply list only their own skill (`/analyze-react`,
`/build-acceptance-criteria`). `taskstoissues`, if a template still declares it, is skip-guarded
(the React flow has no tracker). `gate` maps from the template's `approval` (`human` → `human`,
`auto`/`skip` → `auto`; `implement` is always `human` — the bundle's post-implement code check).
`exitWhen` is one line from the phase map (React exitWhen additions folded in). Global disciplines
(changed tests only, single non-parallel suite run, shared-component immutability, no PR closing
keywords) live in the drivers' Hard Rules.

## Command set

The `sflow/commands/` directory holds 8 stack-neutral `/sf-*` commands (the `sf-` prefix keeps
them distinct from any project-local `/spec-*` set). Mapping from template phase ids:

| Template phase | Command | Notes |
|---|---|---|
| — (scaffold) | `/sf-init` | writes `.specflow/specs/<name>/` + `.meta.yaml` (feature phase list embedded); not a phase |
| preflight | `/sf-preflight` | reuse scan; shared-unit impact; optional design decompose |
| requirements | `/sf-requirements` | stable AC IDs + observable Given/When/Then phrasing |
| clarify | *(none)* | driver-led: one batched Q&A round on the OPEN `## Clarifications` entries |
| design | `/sf-design` | `design.md` + `contracts/` + `qa-journey-plan.md` (when any AC is journey-level); architecture gate PASS or justified |
| tasks | `/sf-tasks` | one test task per AC + edge-case markers |
| taskstoissues | *(none)* | skip-guarded — the React flow has no tracker → `skipped` |
| implement | `/sf-implement` | independent test/impl agents; "completed" ⇒ AC-traceable test green |
| spec-qa | `/sf-qa` | QA audit → `qa-report.md` → human sign-off |
| analysis, describe | *(none)* | driver-led (bugfix/brownfield analysis, quickfix describe) |
| — (pre-gate) | `/sf-validate` | static machine gate; gates `spec-qa`'s `exitWhen` — never a ledger phase |

Commands are **stack-neutral procedure guides**: they describe what each phase does and its
exit gate — no skill names, no driver knowledge — so users can invoke them one by one, and
both drivers can share the same procedure through their own command set (`/sf-*` vs the
project `/spec-*`) to compare performance and quality. Skills are bound to phases only by
the drivers/generators.

`/sf-validate` (static, cheap) runs before `/sf-qa` (expensive, human-dispositioned): a spec
that fails the machine gate never burns QA effort. After `/sf-implement` every workflow stops
for a **human code check** before validate/spec-qa.

## Stack profiles — `react/` and `flutter/`

Two sibling profile directories live alongside `sflow/`:

```
react/
  skills/   build-acceptance-criteria  design-react-architecture  plan-react-tasks
            implement-react-code  build-react-e2e  analyze-react
            oac-figma-decompose  scan-resource
  rules/    architecture-principles.md  test-quality.md

flutter/
  skills/   fl-acceptance-criteria  fl-architecture-design  fl-task-design  fl-implementation
            fl-pr-review  fl-riverpod  fl-test-contract  fl-test-forensics
  rules/    architecture-principles.md  test-quality.md
```

Responsibilities split cleanly: the **generators** own the phase→skill binding table and the
exitWhen lines, and emit the generated `workflow.yaml`; the **drivers** are
pure orchestrators — they run Setup, drive the phases, enforce gates, run the driver-led phases
(bugfix/brownfield `analysis`, quickfix `describe`), and delegate phase work to subagents with
explicit context. No command or domain skill references anything outside its profile by name.

- **React** (`react/`): React 19 + Vite + TypeScript + Zustand + TanStack Query + MUI + Vitest.
- **Flutter** (`flutter/`): Flutter/Dart, state-management-package agnostic. The `fl-*` skills
  exist, but there is no Flutter generator yet — a parallel `*-flutter-workflow` (not shipped)
  would bind them to phases the way the React generators bind the OAC skills, appending `fl-riverpod`
  when `pubspec.yaml` lists Riverpod.

## How to drive a spec

**Recommended:** invoke a driver agent — it runs Setup, waits for your instructions, generates
`workflow.yaml`, then drives every phase in order, pausing at human gates.

| Project | Driver | Commands driven |
|---|---|---|
| Plain (no company toolchain) | `sflow-driver` | `/sf-*` |
| specflow-managed (`.claude/commands/spec-*.md` installed) | `oac-specflow-driver` | the project's `/spec-*` only |

**Ad-hoc:** run `/sf-*` stage commands directly. The commands are stack-neutral; you supply the
stack context manually (or the ambient rules + skills provide it).

## specflow interop

The bundle's spec model is the company specflow toolchain's: same spec dir
(`.specflow/specs/<slug>/`), same artifact names, same `.meta.yaml` shape and status enum
(`pending | in_progress | completed | skipped | failed`), same template phase ids. In a
**specflow-managed** project use **oac-specflow-driver**: everything it runs during phases is a
project `/spec-*` command — the project's `/spec-init` scaffolds (branch + spec dir +
`.meta.yaml`), `/spec-react-workflow` writes `workflow.yaml` (command = `/spec-<id>` per
phase; never touches `.meta.yaml`; no injected phases — `/spec-validate` gates `spec-qa`'s
`exitWhen`, `/spec-drift` is an optional post-merge follow-up). The full contract and drive
rules live in `skills/spec-react-workflow/references/specflow-contract.md`.

## Keeping assets consistent — the editing rules

Apply these whenever an asset is added or changed; every conflict found so far violated one:

- **One producer per fact.** Phase order/approvals live only in the project-hosted templates;
  phase→skill binding tables only in the generators' `references/`; `.meta.yaml` is
  scaffolded only by `/sf-init` (specflow: the project's `/spec-init`); `workflow.yaml` is
  written only by the flow's generator. Never restate another asset's fact — point at it.
- **Consumers name the producer's exact signal** — `workflow:`, `completed`, `<spec-dir>`,
  `gate: human`, `exitWhen`. No synonyms (`complete`, "validated build").
- **Commands are procedure guides — no skill names, no driver knowledge; domain skills and
  rules know nothing about sflow.** Skill↔phase binding lives only in the drivers'
  playbooks and the generators.
- **Artifact requirements are per-workflow**, never a fixed list — the lighter workflows have
  `analysis.md`/`describe.md`, not `requirements.md`+`design.md`.
- **A field nothing reads is deleted**, not kept "for later".
- **Agents may restate a global rule** (engineering-discipline) for subagent copying, but copy it
  verbatim or reference it — never reword it.

## Shared stack-agnostic rules

`engineering-discipline.md` and `preferences.md` are the single source of truth for cross-stack
coding discipline and delegation preferences. They live as **real files** in the repo's top-level
`/rules/` directory only — the profile `rules/` directories carry no copies or symlinks of them.
Do not edit or duplicate them inside a profile.
