# implement

Execute implementation tasks

---

You are an implementation agent for the specflow framework.

## Step 0 — Link the spec to a Jira issue (first run only)

Read `.specflow/specs/<name>/.meta.yaml`. If a `jira_issues:` (or legacy `jira_stories:`) field already exists, skip this step.

Otherwise, follow the procedure in `.claude/commands/_jira-status-automation.md` → **"How issues get associated with a spec"**:

1. Scan `requirements.md`, `clarify.md`, `design.md`, `tasks.md` for `ACMT-\d+` references — surface as suggestions, never auto-accept.
2. Ask the user which existing Jira issue (or issues — 1:many is allowed but 1:1 is expected) this spec maps to. Both **Story** and **Bug** issue types are accepted. At least one key required; abort if user provides none.
3. For each key, call `getJiraIssue({ cloudId: "8c1ec6ab-a1e9-4152-b46f-64eb8df5ea18", issueIdOrKey })` and verify `issuetype.name in {"Story", "Bug"}`. Abort cleanly on Epic / Sub-task / Task / unknown:
   > "`<KEY>` is type `<TYPE>`. Specflow only attaches to product-written Stories or Bugs in ACMT — pick a different issue, or ask product to break Epic-level work into Stories first."
4. Append the validated keys to `.meta.yaml`:
   ```yaml
   jira_issues:
     - ACMT-86   # Story or Bug
   ```

## Step 1 — Transition each issue to `Dev In Progress`

For each key in `.meta.yaml`'s `jira_issues:` (falling back to legacy `jira_stories:` if present), apply the `transitionIssue` primitive from `.claude/commands/_jira-status-automation.md` with target `dev`. The primitive is monotonic and hands-off-aware — it no-ops if the issue is already past `Dev In Progress` or sitting in a frozen status like `Blocked`.

## Step 2 — Implement

Read the `.specflow/specs/<name>/phases.md` generated by `spec:phase`. For each phase in dependency order, execute the phase's incomplete tasks as described below and following the phase's design and subagent coordination. After completing all tasks in a phase, mark the phase as completed in `phases.md` before moving to the next phase.

1. Read the task description and requirements.
2. Before modifying ANY file in a shared component directory, check whether the component is adopted (imported by files outside the shared directory). If adopted, do NOT modify it — instead, create a copy in the domain directory. If unadopted, proceed with the modification.
3. Implement the changes described.
4. Run any relevant tests.
5. Update the task status in tasks.md to "completed".
6. Update the phase status in phases.md to "partially completed" or "completed" as appropriate.

Reference the design in `.specflow/specs/<name>/design.md` for architecture guidance.
References the phases in `.specflow/specs/<name>/phases.md` for subagent coordination and handoff criteria.
References the legacy knowledge base in `.specflow/specs/<name>/references/` for any relevant information discovered during earlier phases.
Reference steering files for coding conventions and project structure.
